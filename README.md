# SOC-Analysis-Project2. Abstract
A Security Operations Center (SOC) is the centralized function within an organization that continuously monitors, detects, analyzes, and responds to cybersecurity incidents. This project simulates a mini-SOC environment using open-source tools to collect logs from various sources (Windows/Linux endpoints, firewalls, web servers), centralize them in a SIEM (Security Information and Event Management) platform, apply detection rules to identify malicious activity, generate alerts, and visualize findings through dashboards. The project covers the complete SOC analyst workflow: log collection, normalization, correlation, threat detection (e.g., brute-force attacks, port scans, malware signatures), alert triage, and incident report generation. The goal is to give a hands-on, end-to-end understanding of how real-world SOC teams detect and respond to cyber threats, bridging theoretical cybersecurity knowledge with practical blue-team operations.

3. Objectives

Understand the role and workflow of a SOC analyst (Tier 1/Tier 2 operations).
Set up a centralized log collection and monitoring pipeline.
Configure a SIEM tool to ingest, parse, and correlate logs.
Detect common attack patterns (brute force, port scanning, malware execution, privilege escalation).
Create detection rules and trigger real-time alerts.
Build a SOC dashboard for visualizing security events.
Perform incident triage and write a professional incident report.
Recommend future improvements (automation, threat intelligence integration).

6. Dataset
Since this is a hands-on lab project, logs are self-generated by simulating attacks in a controlled environment. Two options:
Option A — Live Lab (recommended):
Set up Kali (attacker) + Windows/Linux (victim) VMs on the same network and generate real logs by performing actual attacks (Nmap scan, SSH brute force via Hydra, etc.).
Option B — Public datasets (for additional analysis/practice):

Using a combination of live-generated logs (for your "implementation" originality) and a public dataset (for broader pattern analysis) gives the strongest project.

7. Implementation Steps
Phase 1: Environment Setup

Install VirtualBox/VMware.
Create 3 VMs: Kali Linux (attacker), Windows 10/Ubuntu (victim/endpoint), Ubuntu Server (SIEM host — Wazuh manager).
Configure all VMs on the same internal/host-only network.

Phase 2: SIEM Deployment

Install Wazuh Manager + Indexer + Dashboard on the Ubuntu server (single-line install script available on Wazuh docs).
Install Wazuh Agent on the Windows and Linux victim machines.
Verify agent connectivity in the Wazuh dashboard (Agents tab shows "Active").

Phase 3: Log Source Configuration

Enable Windows Event Logging (Security, System logs) on the Windows victim.
Install Sysmon on Windows for detailed process/network logging.
Enable Auditd on Linux victim for system call logging.
(Optional) Deploy DVWA on a web server VM and enable Apache access/error logs.

Phase 4: Attack Simulation

From Kali, run:

nmap -sS -A <victim-ip> → port scan
hydra -l admin -P rockyou.txt ssh://<victim-ip> → brute force
Upload a malicious test file (EICAR) to simulate malware detection
SQL injection attempt on DVWA


Capture timestamps of each attack for later correlation in reports.

Phase 5: Detection Rule Configuration

Use Wazuh's built-in rule sets (e.g., rule ID 5710 for SSH brute force).
Write a custom rule in local_rules.xml, e.g., to flag >5 failed logins in 60 seconds from the same IP.
Test rule triggers by re-running the attack and confirming alerts appear.

Phase 6: Alerting

Configure Wazuh to send email/Slack alerts for high-severity events (level ≥ 12).
Categorize alerts by severity: Low, Medium, High, Critical.

Phase 7: Dashboard Creation

In Wazuh Dashboard (Kibana-based), build visualizations:

Top attacking IPs
Alert severity distribution (pie chart)
Timeline of events
Geo-map of attacker IPs (if using public IPs/dataset)


Combine into a single SOC Dashboard view.

Phase 8: Incident Response & Reporting

Triage each alert: false positive vs. true positive.
Document each confirmed incident using a structured incident report template (see Section 10).

Jun 28 14:02:19 victim sshd[2214]: Failed password for admin from 192.168.1.15 port 51330 ssh2
Jun 28 14:02:21 victim sshd[2215]: Accepted password for admin from 192.168.1.15 port 51332 ssh2
Analyst Interpretation:

5 failed login attempts within 10 seconds from a single source IP (192.168.1.15), followed by a successful login.
Pattern matches classic brute-force attack behavior.
Source IP should be flagged as malicious; account "admin" may be compromised.

Example — Nmap Port Scan Log (Wazuh alert excerpt):
json{
  "rule": {"id": "100002", "description": "Multiple port scan attempts detected", "level": 10},
  "src_ip": "192.168.1.15",
  "dst_ip": "192.168.1.20",
  "ports_scanned": 1000,
  "timeframe": "15 seconds"
}
Analyst Interpretation:

Reconnaissance activity (port scanning) typically precedes an exploitation attempt — should be flagged and the source IP monitored/blocked.

12. Conclusion
This project provided practical, end-to-end exposure to SOC operations — from log collection and SIEM configuration to threat detection, alerting, and incident reporting. By simulating real attacks (brute force, port scanning, malware execution) in a controlled lab and detecting them using a SIEM platform, the project demonstrates the core responsibilities of a SOC analyst: monitoring, detection, triage, and response. It reinforces the importance of centralized logging, correlation rules, and timely alerting in defending against cyber threats, and builds a strong practical foundation for a career in blue-team/SOC roles.
